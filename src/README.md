# 제품 구현 구성

이 디렉터리는 구현 언어나 라이브러리를 정하지 않고 번역과 이진 배치의 책임을 나눈다.

- `translation/`은 원본에 묶인 번역 자산을 읽고, 안정 ID·보호 정보·제어 토큰·상태·빌드 선택을 검증한 뒤 선택된 단위를 대상 인코딩으로 만든다. 대상 이미지의 주소를 고르거나 직접 쓰지 않는다.
- `placement/`는 인코딩된 단위와 검증된 원본 구조를 받아 저장 위치, 정렬, 패딩, 포인터와 관련 메타데이터 변경을 결정한다. 번역 문장을 고치거나 검토 상태를 바꾸지 않는다.

## 산출물 생명주기

```text
번역 색인·조각
  → 검증된 단위 선택
  → 인코딩된 단위
  → 배치 계획
  → 전역 Expected Write 계획
  → 적용·전체 diff 감사
```

이는 작업 일정을 고정하는 순서가 아니라 산출물의 의존 방향이다. 번역, 폰트, 인코딩과 배치는 조사 결과에 따라 오갈 수 있고 서로 독립적인 범위는 병렬로 진행할 수 있다. 개발 빌드는 미완료 단위를 원문으로 유지할 수 있지만, 어느 경로도 뒤 단계의 임시 결과를 앞 단계의 원천으로 삼지 않는다.

각 경계는 명시적인 입력과 출력을 가지며, 파일 자동 탐색이나 첫 일치로 대상을 고르지 않는다. 인코딩 결과에는 안정 ID와 바이트 길이를, 배치 계획에는 대상 범위와 원본 전제를 남긴다. 용량이나 정렬 조건을 만족하지 못하면 번역을 몰래 줄이거나 자르지 않고 실패 이유를 앞 단계에 돌려준다.

배치 결과는 제품 그래프 안에서 매번 다시 만드는 파생물이다. 커밋된 원천이나 PoC 산출물로 대체하지 않는다. 모든 배치가 모인 전역 Expected Write 계획이 겹침, 영역, 원본 바이트와 기계어 규칙을 통과한 뒤에만 최종 이미지에 적용한다.

`tests/translation/`은 상태·선택·보호 정보·인코딩 경계를, `tests/placement/`는 정렬·용량·주소·포인터·쓰기 소유권을 검증한다. 둘의 연결과 최종 diff 감사는 제품 전체 테스트에서 확인한다.
