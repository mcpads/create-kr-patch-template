# 제품 구현 구성

이 디렉터리는 번역 인코딩, 글리프 변환과 이진 배치의 책임을 나누는 선택 골격이다. 기존 구현에 같은 경계가 있으면 이 디렉터리 구성을 따로 만들지 않는다.

- `translation/`은 원본에 묶인 번역 자산을 읽고, 현재 알려진 안정 ID 모집단과 단위의 일치·보호 정보·제어 토큰·상태·빌드 선택을 검증한 뒤 프로젝트가 명시한 문자 매핑으로 선택된 단위를 인코딩한다. 대상 이미지의 주소를 고르거나 직접 쓰지 않는다.
- `fonts/`는 선택한 기성 글꼴 원천·국소 보완과 같은 문자 매핑을 명시 입력으로 받아 대상 형식의 글리프를 만든다. 글꼴·매핑·보완 범위를 자동으로 선택하지 않고, 대상 이미지에 직접 쓰지 않는다.
- `placement/`는 인코딩된 단위와 검증된 원본 구조를 받아 저장 위치, 정렬, 패딩, 포인터와 관련 메타데이터 변경을 결정한다. 번역 문장을 고치거나 검토 상태를 바꾸지 않는다.

## 산출물 생명주기

```text
번역 색인·조각
  + 명시 문자 매핑
  → 검증·인코딩된 단위
글꼴 원천·국소 보완
  + 명시 문자 매핑
  → 대상 형식 글리프
인코딩된 단위·대상 형식 글리프
  → 배치 계획
  → 전역 Expected Write 계획
  → 적용·전체 diff 감사
```

이는 작업 일정을 고정하는 순서가 아니라 산출물의 의존 방향이다. 번역, 글꼴, 인코딩과 배치는 조사 결과에 따라 오갈 수 있고 서로 독립적인 범위는 병렬로 진행할 수 있다. 개발 빌드는 미완료 단위를 원문으로 유지할 수 있지만, 어느 경로도 뒤 단계의 임시 결과를 앞 단계의 원천으로 삼지 않는다.

각 경계는 명시적인 입력과 출력을 가지며, 파일 자동 탐색이나 첫 일치로 대상을 고르지 않는다. 인코딩 결과에는 안정 ID와 바이트 길이를, 배치 계획에는 대상 범위와 기대 원본 바이트를 남긴다. 용량이나 정렬 조건을 만족하지 못하면 번역을 몰래 줄이거나 자르지 않고 실패 이유를 앞 단계에 돌려준다.

글리프 자산을 바꾸는 프로젝트에서는 글리프 변환이 번역 인코딩과 같은 명시 문자 매핑을 사용하게 하고, 둘을 배치와 같은 주 빌드에 연결한다. 글리프만 따로 만든 이미지나 텍스트만 따로 넣은 이미지를 전체 구현의 성공으로 보지 않는다.

배치 결과는 주 빌드에서 매번 다시 만드는 파생물이다. 커밋된 원천이나 PoC 산출물로 대체하지 않는다. 모든 배치가 모인 전역 Expected Write 계획이 겹침, 영역, 원본 byte와 기계어 규칙을 통과한 뒤에만 최종 이미지에 적용한다.

`tests/translation/`은 상태·선택·보호 정보·인코딩 경계를, `tests/fonts/`는 원천 식별·글리프 커버리지·문자 매핑·변환 재현성을, `tests/placement/`는 정렬·용량·주소·포인터·쓰기 주체를 검증한다. 필요한 경계의 연결과 최종 diff 감사는 제품 전체 테스트에서 확인한다.
