# Repository Guidelines

## 시작할 때

- 기존 저장소의 빌드·테스트·문서·자산 흐름을 먼저 확인한다. 동등한 구조가 있으면 이 골격을 병설하지 않는다.
- `.gitignore`가 실제 원본·출력·작업·실행 증거 경로를 덮는지 먼저 확인한다.
- `reference/rust/`는 선택 가능한 참고 구현이다. 대상 프로젝트에 이미 같은 검사가 있으면 복사하지 않는다.

## 반드시 자동 실패시킬 것

- 원본 크기·강한 해시 불일치
- Expected Write의 원본 byte 불일치, 쓰기 겹침, 보호·비대상 영역 침범과 미등록 최종 diff
- 데이터 쓰기를 통한 기계어 변경
- 필요한 어셈블리 원천과 대상 ISA 검산이 없는 기계어 생성·이동
- 선언한 무변경 경계의 byte 차이
- 실패 뒤 정상 산출물처럼 보이는 부분 출력

모든 최종 쓰기는 불변 원본을 기준으로 검증한 뒤 한 번에 적용한다. 여러 변경이 같은 범위를 의도적으로 만들면 적용 순서에 맡기지 않고 하나의 쓰기 결과로 먼저 합성한다.

## 프로젝트에서 판정할 것

- 번역 모집단 완전성, 대표성, 사람 승인, UX와 실행 범위는 자기 선언만으로 범용 자동 판정을 만들지 않는다.
- 대상 조사로 확정한 누락·범위·인코딩·길이·소비 조건은 프로젝트 테스트로 고정한다.
- 제품 빌드는 원본과 버전 관리되는 입력에서 다시 만들고, PoC 출력이나 수동 파생물을 읽지 않는다. 범용 그래프 매니페스트보다 실제 주 빌드의 깨끗한 재실행으로 확인한다.
- 채택한 기술 변경은 미완료 번역을 원문으로 남기더라도 하나의 개발 빌드에 통합한다.
- 실행 결과는 정확한 산출물에 귀속하고, 관측 범위와 HITL 판단을 해시 일치로 대신하지 않는다.

## 조사와 구현

- 조사와 PoC의 언어·도구·순서는 현재 질문에 맞게 고른다. 작은 Python 프로그램이 가장 짧으면 사용하되 제품 구현의 기본 언어로 강제하지 않는다.
- 주요 문자 집합은 검증 가능한 기성 글꼴에서 시작하고, 직접 제작은 필수 누락이나 확인된 UX 요구의 국소 보완에 한정한다.
- 자동 검출과 수정 제안은 허용한다. 번역 일괄 변경은 기준선·규칙·범위·예상 영향을 사람이 승인한 뒤 적용한다.
- 서로 독립적으로 바뀌고 시험할 책임은 나누되, 파일 수나 길이 자체를 목표로 삼지 않는다. 테스트는 구현 파일과 분리해 `*_tests.*`처럼 역할이 드러나게 둔다.

## 참고 구현 확인

```bash
cd reference/rust
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --locked
```
