# Repository Guidelines

## 역할

- `conformance/`는 언어와 플랫폼에 독립적인 필수 판정을 소유한다.
- `reference/rust/crates/patch-guard/`는 그 판정의 교체 가능한 Rust 참고 구현이다. 플랫폼 지식이나 특정 도구를 추가하지 않는다.
- `reference/rust/examples/synthetic-cli/`는 대상별 CLI를 확장하는 방법만 보이는 자작 예제다. 그 포맷과 수치를 일반화하지 않는다.
- 실제 프로젝트의 조사 기록과 PoC는 제품 그래프 밖에 둔다. 확인된 변환은 순수 원천에서 재현되는 제품 단계로 다시 구현한다.
- 원본, 전체 추출물과 빌드 산출물은 로컬 전용 경로에 둔다.

## 반드시 지킬 것

- 개발 편의를 위해 원본 신원, Expected Write, 영역 분류, 기계어 검증 또는 제품 입력 계보를 우회하지 않는다.
- 모든 최종 쓰기는 검증된 원본 기준의 전역 계획 하나에 모은다. 각 쓰기의 행위자와 목적은 하나이며, 겹치는 쓰기는 같은 행위자끼리라도 실패시킨다. 의도적인 합성은 쓰기 계획을 만들기 전에 한 행위자의 결과로 정리한다.
- 데이터 쓰기로 기계어 영역을 바꾸지 않는다. 기계어 변경은 원시 바이트 상수가 아니라 어셈블리 원천에서 생성하고, 대상 ISA 검증기로 조립 결과와 전 구간 해석을 확인한다.
- 제품 빌드가 조사 산출물, 수동으로 가공한 중간 파일 또는 등록되지 않은 외부 파생물을 읽게 하지 않는다.
- 무변경 왕복을 주장하는 변환은 일부 필드가 아니라 선언한 전체 경계를 바이트 단위로 비교한다. 알 수 없는 구조는 그대로 보존하거나 거부하며, 다른 직렬화가 불가피하면 그 차이도 Expected Write로 소유한다.
- 휴리스틱은 후보를 찾는 조사에만 쓴다. 확인된 대상은 명시적인 값과 원본 바이트 전제로 고정하며, 제품 빌드가 점수나 첫 일치로 쓰기 위치를 선택하게 하지 않는다.
- 미완료 번역은 개발 빌드를 막는 이유가 아니다. 프로젝트가 원문 유지·초안 사용 범위를 명시하고 보고서에 남긴다. 릴리스 후보에는 선언 범위의 완료와 현재 내용에 묶인 사람 승인을 요구한다.
- 자동 검출과 수정 제안은 허용한다. 규칙 기반 일괄 수정은 사람이 기준선, 규칙, 범위와 예상 영향을 승인한 뒤에만 적용한다.
- 빌드 성공, 릴리스 적격성과 실행 검증을 같은 상태로 표현하지 않는다. 실행 통과는 정확한 빌드 결과와 독립 증거의 해시가 일치할 때만 인정한다.
- 모든 검사를 통과한 메모리 결과만 출력 경로에 기록한다. 실패한 빌드가 부분 산출물을 정상 결과처럼 남기지 않게 한다.
- 런타임 검증에 `emucap`을 선택하면 `adapters/emucap/README.md`를 먼저 읽는다. 설치와 에뮬레이터별 절차는 공개 배포 문서를 따르고, 이 저장소에는 산출물과 증거를 연결하는 규칙만 유지한다.

## 확장 방식

- 새 CLI와 대상 어댑터는 자유롭게 추가한다. 모든 빌드 진입점은 같은 필수 판정을 호출해야 한다.
- 데이터 형식, 디렉터리, 라이브러리와 명령 이름은 대상에 맞게 바꿀 수 있다. `conformance/`가 요구하는 의미만 보존한다.
- 런타임 관찰 도구는 교체할 수 있다. 어떤 도구를 쓰더라도 정확한 산출물 식별, 시나리오 결과, 증거 식별과 상태 변경 개입 기록을 보존한다.
- 플랫폼별 예외가 필요해 보이면 참고 코어를 완화하기 전에 대상 모델이나 검증기를 더 정확히 정의한다.
- 반례를 새로 발견하면 구현 한 곳에만 고치지 말고 언어 중립 반례와 참고 구현 테스트를 함께 추가한다.

## 조사와 PoC

- 기존 저장소에 검증된 분석 기반이 있으면 그 구조를 유지한다. 새 조사 기반이 필요하면 `research/poc/`의 작은 Python 프로그램을 기본 선택으로 삼되, 대상 도구와 직접 연동해야 하거나 다른 수단이 더 짧게 가설을 가르면 그 수단을 선택할 수 있다.
- PoC 하나는 현재 판단을 바꿀 질문 하나를 다룬다. 입력과 출력은 명령행 인자로 받고, 입력 신원·관찰 조건·결과를 남겨 다른 에이전트가 같은 실험을 다시 실행할 수 있게 한다. 셸 한 줄이나 대화형 세션에만 남은 조작은 증거로 취급하지 않는다.
- 유지할 가치가 있는 파서와 변환은 작은 함수로 나누고 같은 위치의 `*_tests.py`에서 자작 입력으로 검증한다. 임시 의존성을 제품 구현에 전파하지 않는다.
- PoC 프로그램과 재현 절차는 커밋할 수 있지만, 그 산출물은 `out/research/`, `work/`나 비공개 증거 경로에 둔다. 제품 빌드는 PoC 산출물을 읽지 않는다.
- PoC에서 확정한 변환이 제품에 필요해지면 순수 원천에서 다시 만드는 제품 단계로 구현한다. Python 프로그램 자체를 제품 단계로 승격한다면 `research/` 밖으로 옮기고 다른 제품 구현과 같은 입력 계보·Expected Write·검증을 적용한다.

## 파일 구성

- 새 기능을 가장 큰 기존 파일에 덧붙이는 것을 기본 선택으로 삼지 않는다. 입력 형식, 변환, 검증, 보고처럼 독립적으로 바뀌고 시험할 수 있는 책임은 이름이 드러나는 별도 모듈이 소유한다.
- 진입점은 인자 해석과 조립만 담당한다. 대상 포맷, 인코딩, 배치, 쓰기 계획과 검증 구현을 진입점에 누적하지 않는다.
- 파일 길이만 맞추려고 임의로 나누지 않는다. 한 파일에 서로 다른 변경 이유가 생기거나 일부만 독립적으로 교체·시험할 수 있게 되면 그 책임 경계에서 나눈다.
- `utils`, `helpers`, `manager`처럼 서로 무관한 기능을 받아들이는 포괄 모듈을 만들지 않는다. 공통 코드도 실제 책임의 이름으로 둔다.
- Rust 단위 테스트는 구현 파일 안에 쓰지 않고 같은 디렉터리의 `*_tests.rs`에 둔다. 통합 테스트 파일도 `*_tests.rs`로 끝나야 하며, 참고 구현의 구조 검사가 이를 확인한다.
- 번역 자산의 파일 구성은 `assets/translation/README.md`가 소유한다. 하나의 계속 커지는 파일이나 명시되지 않은 파일 자동 탐색을 제품 입력으로 삼지 않는다.
- 번역 선택·인코딩과 이진 배치의 구현 경계는 `src/README.md`를 따른다. `tests/translation/`과 `tests/placement/`에서 각 경계를 따로 검증하고, 전체 제품 그래프 검증은 둘을 함께 통과시킨다.

## 확인 명령

```bash
python3 -m unittest discover -s reference/python/poc -p '*_tests.py'
cd reference/rust
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --locked
```
