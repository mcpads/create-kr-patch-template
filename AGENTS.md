# Repository Guidelines

## 시작할 때

- 기존 저장소에서는 이미 쓰는 빌드·테스트·문서·자산 흐름을 유지한다.
- 새 프로젝트의 README는 대상 게임, 지원 원본, 현재 범위와 빌드 진입점을 중심으로 작성하고, 작업 규칙은 `AGENTS.md`에 둔다.
- `.gitignore`가 실제 원본·출력·작업·실행 증거 경로를 덮는지 먼저 확인한다.
- `reference/rust/`는 선택 가능한 참고 구현이다.

## 반드시 자동 실패시킬 것

- 원본 크기·강한 해시 불일치
- Expected Write의 원본 byte 불일치, 쓰기 겹침, 보호·비대상 영역 침범과 미등록 최종 diff
- 데이터 쓰기를 통한 기계어 변경
- 필요한 어셈블리 원천과 선언한 대상 ISA 프로필의 조립·해석·재조립 검산이 없는 기계어 생성·이동
- 선언한 무변경 경계의 byte 차이
- 실패 뒤 정상 산출물처럼 보이는 부분 출력

모든 최종 쓰기는 불변 원본을 기준으로 검증한 뒤 한 번에 적용한다. 여러 변경이 같은 범위를 의도적으로 만들면 적용 순서에 맡기지 않고 하나의 쓰기 결과로 먼저 합성한다.

## 프로젝트에서 판정할 것

- 번역 모집단 완전성, 대표성, 사람 승인, UX와 실행 범위는 조사와 HITL로 판정한다. 자동 검사는 대상에서 확정한 누락·범위·인코딩·길이·소비 조건에 한정한다.
- 제품 빌드는 원본과 버전 관리된 입력만으로 처음부터 다시 만들며, PoC 출력과 수동 파생물은 제품 입력에서 제외한다.
- 채택한 기술 변경은 미완료 번역을 원문으로 남기더라도 하나의 개발 빌드에 통합한다.
- 실행 결과는 정확한 산출물에 귀속하고, 관측 범위와 HITL 판단을 해시 일치로 대신하지 않는다.

## 조사와 구현

- 조사와 PoC의 언어·도구·순서는 현재 질문에 맞게 고른다. 작은 Python 프로그램이 가장 짧으면 사용하되 제품 구현의 기본 언어로 강제하지 않는다.
- 주요 문자 집합은 검증 가능한 기성 글꼴에서 시작하고, 직접 제작은 필수 누락이나 확인된 UX 요구의 국소 보완에 한정한다.
- 자동 검출과 수정 제안은 허용한다. 번역 일괄 변경은 기준선·규칙·범위·예상 영향을 사람이 승인한 뒤 적용한다.
- 서로 독립적으로 바뀌고 시험할 책임은 나눈다.
