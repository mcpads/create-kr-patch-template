# emucap 런타임 검증 어댑터

이 어댑터는 빌드한 패치가 실제 실행 경로에서 소비되는지 확인할 때 [emucap](https://github.com/mcpads/emucap) 사용을 권장한다. `conformance/runtime-evidence.json`의 판정은 도구와 무관하며, 같은 증거 요건을 만족하는 다른 관찰 수단으로 바꿀 수 있다.

`emucap`의 설치법, 지원 플랫폼과 에뮬레이터별 절차는 이 저장소에 복제하지 않는다. 에이전트가 공개 배포 저장소의 README에 따라 공통 구성요소를 준비하고, 현재 대상에 필요한 어댑터만 설치한다. 사용자가 터미널 설치 절차를 직접 수행한다고 가정하지 않는다.

## 연결 규칙

1. 검증할 빌드 산출물의 SHA-256을 빌드 보고서에서 확정한다. 같은 파일의 SHA-1도 계산해 `emucap`이 보고하는 ROM 식별 정보와 대응시킨다. 추적 원장의 SHA-1만으로 이 템플릿의 산출물 식별을 대신하지 않는다.
2. 제어 MCP와 추적 MCP의 `bootstrap`을 각각 호출한다. 로컬 설치 경로, 포트와 지원 기능을 추측하지 않는다.
3. 제어 MCP의 `get_rom_info`가 반환한 `rom_sha1`과 현재 연결의 `connection_ref`를 추적 MCP의 `run_start`에 전달한다. 시작 전에 계산한 SHA-1과 다르면 검증을 중단한다.
4. 시나리오마다 자연 실행 경로, 시작 조건과 기대 관찰을 정한다. `write_memory`, 상태 불러오기, 리셋, 입력 주입처럼 실행 상태를 바꾼 개입은 `log_intervention`으로 남기고, 개입한 실행을 자연 도달 증거처럼 표현하지 않는다.
5. 제어 MCP의 분석 결과는 자동으로 원장에 기록된다고 가정하지 않는다. 오래 보존할 판정과 측정값은 추적 MCP에 명시적으로 기록한다.
6. 실행이 끝나면 정확한 산출물 ID와 SHA-256, 시나리오 결과, 독립 증거 파일의 ID와 SHA-256으로 `conformance/runtime-evidence.json`의 요건을 만족하는 보고서를 만든다. 빌드 보고서의 성공 상태를 수정하는 대신 별도 증거로 연결한다.

화면 캡처, 추적 로그와 원장의 원본은 조사 증거이지 제품 빌드 입력이 아니다. 저작권이 있거나 로컬 경로·환경 정보가 포함될 수 있는 자료는 `evidence/private/`와 `.emucap/`에 두고, 공개 저장소에는 배포할 수 있는 요약과 식별자만 남긴다.

## 연결 이상과 종료

시간 초과나 연결 끊김만으로 에뮬레이터가 종료됐다고 판단하지 않는다. 공개 배포 문서가 안내하는 연속성과 실행 인스턴스를 먼저 확인하고, 살아 있는 세션에는 다시 연결한다.

검증을 마친 뒤 에뮬레이터를 종료해야 한다면 현재 연결에 속한 실행 인스턴스와 검증 대상 경로를 확인한 뒤 그 프로세스만 종료한다. 이름이 같은 프로세스를 한꺼번에 종료하지 않는다.
