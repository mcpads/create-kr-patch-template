# 레트로 게임 한글 패치 프로젝트 템플릿

플랫폼과 구현 언어를 정하기 전에, 패치 프로젝트가 지켜야 할 검증 경계를 먼저 세우는 시작점이다. 플랫폼 정보, 파일 포맷, 폰트, 압축, 패치 형식과 도구 선택은 대상 조사 뒤 프로젝트가 정한다.

이 저장소의 Rust 코드는 필수 구현이 아니라 교체 가능한 참고 구현이다. 다른 언어로 바꾸더라도 [`conformance/`](conformance/)의 반례를 같은 결과로 판정하면 된다. 한글화 판단 방법은 [create-retro-game-kr-patch](https://github.com/mcpads/create-retro-game-kr-patch)가 설명하며, 이 템플릿은 반복 빌드에서 반드시 막아야 할 실패를 실행 가능한 형태로 제공한다.

## 고정하는 경계

- 지원 원본은 크기와 강한 해시로 식별한다.
- 각 최종 이진 산출물의 모든 변경은 검증된 원본을 기준으로 한 번의 전역 계획에 모으고, 적용 전에 쓰기 주체, 목적, 기대 원본과 대상 영역을 선언한다.
- 서로 겹치는 쓰기, 보호 영역 쓰기, 기대 원본 불일치와 미등록 최종 변경은 항상 실패한다.
- 기계어 영역은 데이터 쓰기로 바꿀 수 없다. 어셈블리 소스와 ISA 프로필을 실제로 대조하는 프로젝트 검증기가 없으면 기계어 쓰기도 실패한다.
- 제품 빌드의 시작 입력은 순수 원천뿐이다. 조사·PoC 산출물이나 파이프라인 밖에서 미리 가공한 파생물을 제품 입력으로 연결하면 실패한다.
- 제품 단계가 만든 파생물은 같은 제품 그래프 안에서 다시 만들 수 있어야 하며, 각 산출물을 만드는 단계는 하나여야 한다.
- 본 구현에 채택한 변경은 같은 개발 빌드 결과에 합쳐야 한다. 구성요소별 성공을 합산해 전체 구현의 성공으로 보지 않는다.
- 무변경 왕복을 주장하는 파서·코덱·재조립 단계는 선언한 전체 경계를 바이트 단위로 재현해야 한다.
- 개발 빌드는 진행 중인 모집단 조사와 미완료 번역을 허용하되, 현재 알려진 단위는 모두 원문 유지 또는 번역 사용 상태로 다룬다. 릴리스 후보만 선언 범위의 모집단 확정, 전체 단위의 완료와 현재 범위·내용에 대한 사람 승인을 요구한다.
- 주요 문자 집합은 검증 가능한 기성 글꼴에서 공급한다. 직접 제작·보정은 필수 글리프의 국소 누락이나 확인된 UX 요구에 한정한다.
- 빌드 보고서는 모든 원천 입력과 결과를 강한 해시로 식별한다. 실행 성공은 빌드 상태에 덧붙이는 플래그가 아니라 해당 결과 해시에 묶인 별도 증거로 기록한다.

이 경계 안에서 조사 순서, 실험 횟수, 도구, 데이터 형식과 CLI 명령은 고정하지 않는다. 에이전트는 선언한 목표의 완성을 막을 수 있는 미확정 조건을 먼저 확인한다. 같은 조건을 판별할 방법이 여럿이면 비용이 낮은 증거부터 얻을 수 있다.

여기서 순수 원천은 사용자가 공급한 원본과 버전 관리되는 번역·설정·소스처럼 제품의 출발점으로 검토할 수 있는 입력이다. `work/`나 `out/`에서 만들어진 파일, 수동 전처리 결과와 PoC 출력은 해시를 붙여도 순수 원천이 아니다.

## 조사와 제품 빌드

조사와 PoC는 자유롭게 실패하고 버릴 수 있는 공간이다. 여기서 얻은 결론을 본 구현에 채택하면 PoC 결과 파일을 복사해 넣지 않고, 순수 원천에서 같은 결과를 다시 만드는 제품 단계로 등록한다. 등록 뒤에는 그 시점까지 채택한 다른 변경과 함께 개발 빌드를 다시 만들어 통합 결과를 확인한다.

```text
순수 원천 ── 제품 단계 ── 제품 단계 ── 최종 이미지
                   └──── 검증 보고서

조사 입력 ── 탐색/PoC ── 관찰·근거
                         ╳ 제품 단계의 입력
```

제품 그래프는 고정된 단계 순서가 아니다. 서로 독립적인 단계는 병렬로 구현할 수 있고, 새 증거가 판단을 뒤집으면 관련 단계만 바꾼다. 다만 어느 경로로 실행해도 최종 산출물은 등록된 제품 그래프와 순수 원천만으로 재생성돼야 한다.

후보 스캐너와 점수 기반 휴리스틱은 조사 도구다. 주소·구조·상수가 확인되면 검토 가능한 대상 명세로 옮기고 원본 바이트로 다시 검증한다. 제품 빌드는 선언된 값을 찾았는지 확인할 수는 있지만, 실행할 때마다 가장 그럴듯한 후보를 골라 쓰기 대상으로 삼아서는 안 된다.

## 저장소 구성

- `assets/`: [번역 조각](assets/translation/)과 [기성 글꼴 원천·국소 보완](assets/fonts/), 직접 작성한 그래픽 원천을 위한 프로젝트 골격
- `config/`: 대상 프로젝트가 채울 설정 골격
- [`src/`](src/): 번역 인코딩·글리프 변환·이진 배치를 분리한 제품 구현 골격
- `tests/translation/`, `tests/fonts/`, `tests/placement/`: 번역·글리프·배치 경계와 그 연결을 검증하는 골격
- [`research/`](research/): 제품 그래프와 분리된 조사 기록과 재현 가능한 PoC 원천
- `conformance/`: 특정 언어에 속하지 않는 필수 판정과 반례
- `adapters/emucap/`: 선택 가능한 런타임 검증 어댑터
- `reference/rust/crates/patch-guard/`: 위 판정을 구현한 Rust 참고 라이브러리
- `reference/rust/examples/synthetic-cli/`: 자작 입력만 쓰는 교체 가능한 CLI 예제
- `reference/python/poc/`: 제품 CLI와 분리된 Python PoC 예제
- `AGENTS.md`: 이 템플릿을 확장하는 에이전트가 지킬 작업 경계

대상 프로젝트는 합성 예제를 복사해 플랫폼 구현을 덧붙여도 되고, 다른 언어로 검증 코어와 CLI를 새로 만들어도 된다. 참고 라이브러리를 수정해 플랫폼 예외를 넣기보다 대상 어댑터와 검증기를 확장하는 편이 경계를 보존하기 쉽다.

## 참고 구현 실행

```bash
cd reference/rust
cargo run -p synthetic-cli -- make-source --output out/demo/source.bin

# 제품 입력으로 쓰이지 않는 Python 조사 예시
python3 ../python/poc/scan_ascii.py \
  --source out/demo/source.bin \
  --evidence out/research/poc.json

# 미완료 항목을 원문으로 남기는 개발 빌드
cargo run -p synthetic-cli -- build \
  --mode development \
  --source out/demo/source.bin \
  --translations examples/synthetic-cli/assets/demo.in-progress.json \
  --output out/demo/development.bin \
  --report out/demo/development-report.json

# 선언 범위가 완료된 릴리스 후보
cargo run -p synthetic-cli -- build \
  --mode release-candidate \
  --source out/demo/source.bin \
  --translations examples/synthetic-cli/assets/demo.complete.json \
  --output out/demo/release.bin \
  --report out/demo/release-report.json

# 같은 입력에서 새로 만든 결과와 바이트 단위 비교
cargo run -p synthetic-cli -- verify \
  --mode release-candidate \
  --source out/demo/source.bin \
  --translations examples/synthetic-cli/assets/demo.complete.json \
  --output out/demo/release.bin
```

`build`에는 PoC 산출물을 받는 옵션이 없다. 합성 예제의 포인터, 인코딩과 체크섬은 자작 포맷을 완주해 보기 위한 예시일 뿐 다른 게임에 전이할 지식이 아니다.

## 대상 프로젝트로 확장할 때

1. 지원 원본과 소비 경로를 조사하고, 선언한 대상 범위의 현재 모집단과 범위 확정 기준을 기록한다. 아직 전체 범위를 확정하지 못했다면 조사 중으로 표시하며, 일부 PoC 추출을 전체로 간주하지 않는다. 대상별 형식·주소·인코딩은 참고 코어 밖에 둔다.
2. 추출·글리프 변환·인코딩·배치·재삽입·무결성 보정 가운데 채택한 변경을 모두 순수 원천에서 다시 만드는 제품 그래프로 연결한다.
3. 파서·코덱·재조립기를 제품 경로에 넣기 전에 무변경 입력을 같은 경로로 통과시켜 전체 경계를 검증한다. 알 수 없는 구조는 그대로 보존하거나 쓰기 전에 거부하고, 바이트 동일성을 보장하지 않는 변환은 그 변경도 쓰기 계획에 포함한다.
4. 번역 자산 형식은 프로젝트에 맞게 정하되 현재 알려진 안정 ID 모집단과 번역 단위의 일치, 보호 원문, 제어 토큰, 선택 상태와 현재 범위·내용에 묶인 릴리스 승인을 검증한다.
5. 글꼴 원천과 보완 글리프는 [`assets/fonts/README.md`](assets/fonts/README.md)에 따라 선택·식별하고, 변환 결과와 명시한 문자 매핑을 같은 제품 그래프에 연결한다.
6. 최종 산출물을 바꾸는 모든 구현을 Expected Write와 전체 diff 감사에 연결한다.
7. 기계어를 바꾼다면 어셈블리 소스, 완전한 ISA 프로필, 조립 결과 일치와 전 구간 디코드 검사를 수행하는 검증기를 추가한다.
8. 필요한 CLI 하위 명령은 자유롭게 추가하되, 우회 명령도 같은 정적 검사를 통과하게 한다.
9. 런타임 자산은 저장됐다는 사실과 실제 소비됐다는 사실을 구분한다. 대상에 필요한 저장·참조·적재·변환·상주·소비 경계를 검증하고, 증거를 정확한 결과 해시에 묶는다.
10. 배포 패치를 다시 적용한 결과와 빌드 산출물의 동일성, 그리고 실제 실행 경로는 빌드와 별도로 검증한다.

런타임 검증에는 [emucap 어댑터](adapters/emucap/) 사용을 권장한다. 이 선택은 필수가 아니며, 정확한 산출물 식별과 독립 증거를 제공하는 다른 관찰 수단으로 교체할 수 있다. `emucap`을 선택했다면 설치·연결 방법을 프로젝트에 복제하지 말고 공개 배포 문서와 이 저장소의 얇은 연결 규칙을 따른다.

원본 이미지와 패치된 전체 이미지는 커밋하지 않는다. 직접 작성했거나 배포 조건을 확인한 입력과 원본을 재현하지 않는 결과만 저장소에 둔다.

## 자체 검증

원격 CI는 기본으로 만들지 않는다. 에이전트가 로컬에서 다음 검사를 실행하며, 외부 실행 비용이 드는 자동화는 프로젝트 사용자가 별도로 선택한다.

```bash
python3 -m unittest discover -s reference/python/poc -p '*_tests.py'
cd reference/rust
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --locked
```
